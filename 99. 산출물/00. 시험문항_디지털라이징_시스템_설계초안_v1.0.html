01

시스템 개요

한글(HWP/HWPX) 파일로 제작된 시험지를 디지털라이징하여, 교사가 메타정보 기반으로 시험지를 구성하고 학생에게 배포·채점·추천까지 제공하는 End-to-End 시스템입니다.

핵심 요구사항

입력 포맷HWP (바이너리) + HWPX (XML) 모두 지원
대상 과목수학 우선 → 전 과목 확장
문항 유형객관식 / 단답형 / 서술형 / 빈칸채우기 등
시험지 양식자체 제작 (양식 통제 가능)
처리 단위문항 1개 단위로 파이프라인 개별 진행
메타 스키마과목별 동적 스키마 관리

04

문항 단위 파이프라인 관리

하나의 HWP 파일에 N개의 문항이 포함되어 있어도, 파싱 이후 각 문항은 개별적으로 파이프라인을 진행합니다.

핵심 원칙: 문항 1개 = 파이프라인 인스턴스 1개. 같은 파일 내에서 통과된 문항은 다음 단계로, 반려된 문항만 재작업 대상이 됩니다. 서로 간섭하지 않습니다.

파일 → 문항 분리 흐름

// 1개 HWP 파일 업로드 대단원평가_2회_문제지.hwp (25문항) │ ├─→ QI-2408-0001-00 → [PARSING] → [PARSE_REVIEW] ✅ → [META] → ... ├─→ QI-2408-0002-00 → [PARSING] → [PARSE_REVIEW] ✅ → [META] → ... ├─→ QI-2408-0003-00 → [PARSING] → [PARSE_REVIEW] ❌ → 반려 → 재작업 → QI-2408-0003-01 ├─→ QI-2408-0004-00 → [PARSING] → [PARSE_REVIEW] ✅ → [META] → ... │ ... └─→ QI-2408-0025-00 → [PARSING] → ...
05

Primary Key 설계

모든 문항은 시스템 전체에서 유일한 식별자를 가집니다. 키만 보고도 출처와 버전을 추적할 수 있어야 합니다.

QI - 2408 - 0031 - 00
접두어 QI
Question Item
소스배치 ID 2408
파일 업로드 시 발번
문항 시퀀스 0031
파일 내 순번 (4자리)
버전 00
원본=00, 수정=01, 02...
버전 관리 규칙: 반려 후 재작업 시 버전이 +1 증가합니다 (00→01→02). 이전 버전은 삭제하지 않고 이력으로 보존하여 추적성을 확보합니다.
06

문항 상태 머신

각 문항은 항상 아래 상태 중 하나에 존재하며, 정해진 전이 규칙에 따라서만 이동합니다.

상태 전이 흐름

PARSING → PARSE_REVIEW → META → META_REVIEW → PRODUCTION → PROD_REVIEW → DATA → DATA_REVIEW → COMPLETED

문항 상태 데이터 구조

07

중복 방지 체계

같은 파일 재업로드, 반려 후 재투입 등으로 인한 문항 중복을 3중으로 방지합니다.

🔒
Content Hash
발문 + 수식 + 선지를 정규화하여 해시 생성. 동일 내용 문항의 이중 투입 차단
📍
Source Tracking
file_hash + position_in_file 조합으로 같은 파일의 같은 위치 문항 재투입 방지
⚡
Pessimistic Lock
문항이 스테이지에 진입하면 locked_by 점유 표시. 동시 작업 충돌 원천 차단
08

검수 기준 총괄

모든 검수팀은 동일한 합격/반려 프레임워크를 따릅니다.

만점
100점
합격 기준
85점 이상
자동 반려 기준
단일항목 50% 미만
자동 반려 룰 (치명적 결함): 어떤 검수 항목이든 해당 항목 배점의 50% 미만 점수를 받으면, 총점이 85점 이상이더라도 무조건 자동 반려됩니다. 이는 핵심 품질 요소에서 치명적 결함이 있는 문항이 파이프라인을 통과하는 것을 원천 차단합니다.

합격/반려 판정 로직

// 검수 판정 Pseudocode function evaluate(scores, criteria): total = 0 for each item in criteria: item_score = scores[item.key] item_max = item.max_score item_rate = item_score / item_max // 치명적 결함 체크: 단일 항목 50% 미만 → 즉시 반려 if item_rate < 0.5: return { result: "REJECT", reason: "AUTO_REJECT_CRITICAL", failed_item: item.key } total += item_score // 총점 기준 판정 if total >= 85: return { result: "PASS", score: total } else: return { result: "REJECT", reason: "BELOW_THRESHOLD", score: total }
