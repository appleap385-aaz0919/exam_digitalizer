바이너리 HWP 파싱 전략v2.1

Phase 1v2.1
v2.0 설계 공백 해소: 샘플이 모두 HWPML이라 간과했지만, 실제 바이너리 HWP가 투입될 경우의 처리 전략.

확정 전략: 변환 후 처리 (Convert-then-Parse)

바이너리 HWP → HWPX 변환 → 기존 XML 파서로 처리. 바이너리 HWP를 직접 파싱하는 것보다 변환 후 처리가 안정적입니다. pyhwp/olefile은 수식·이미지 추출이 불안정하고, 한글 버전별 차이도 큽니다.
포맷 감지 — 매직 바이트 확인: D0 CF 11 E0 → 바이너리 HWP (OLE2)
LibreOffice 변환 — soffice --headless --convert-to hwpx 실행. 변환 실패 시 PARSE_ERROR + 관리자 알림
sandbox 환경 주의: scripts/office/soffice.py 래퍼 사용 (bare soffice는 seccomp 필터에 걸림)
변환 손실 검증 — 변환된 HWPX의 페이지 수, 문항 수, 이미지 수를 원본 메타와 비교. 10% 이상 차이 시 경고
알려진 손실 항목: 일부 특수 글꼴, 한글 전용 OLE 객체, 매우 오래된 수식(Equation 3.0)
HWPX/HWPML 파서로 처리 — 변환된 파일을 기존 XML 파싱 로직에 투입. 이후 동일 파이프라인

직접 파싱 폴백 (변환 실패 시)

// LibreOffice 변환 실패 시 olefile 기반 최소 추출 import olefile ole = olefile.OleFileIO("exam.hwp") // 스트림 구조: // FileHeader → HWP 버전 정보 // DocInfo → 글꼴, 스타일 정보 // BodyText/Section0 → 본문 텍스트 (압축됨, zlib) // BinData/BIN0001.png → 임베딩 이미지 // Scripts/DefaultJScript → 매크로 for stream in ole.listdir(): if "BodyText" in stream: data = ole.openstream(stream).read() text = zlib.decompress(data, -15) // raw deflate // 바이너리 레코드 파싱 필요 (HWPTAG_PARA_TEXT 등) if "BinData" in stream: img_data = ole.openstream(stream).read() // 이미지 파일로 저장
직접 파싱은 텍스트+이미지만 가능. 수식은 별도 바이너리 레코드(HWPTAG_EQEDIT)에 있어 추출이 매우 불안정합니다. 직접 파싱 시에는 수식을 이미지로 캡처하여 fallback_image로 처리합니다.
