CONTEXT

프로젝트 전체 컨텍스트

이 섹션은 설계 과정에서 내려진 모든 결정사항을 요약합니다. 4개 설계 문서와 함께 이 지시서를 참조하면 전체 맥락을 파악할 수 있습니다.

프로젝트 개요

한글(HWP/HWPX) 파일로 제작된 시험지를 AI 에이전트 파이프라인으로 디지털라이징하여, 교사가 메타정보 기반으로 시험지를 구성하고, 학생에게 온라인 CBT로 배포하며, 자동채점 후 유사문항을 추천하는 End-to-End 교육 평가 시스템입니다.

HWPML 구조 핵심

HWPML XML 구조: <HWPML> <HEAD> ← 문서 설정, 폰트, 스타일 <MAPPINGTABLE> <BINDATALIST> ← 이미지 Base64 데이터 <BINDATA Id="1" Encoding="Base64">...</BINDATA> <BODY> <SECTION> <P> ← 문단 <TEXT> <CHAR>텍스트</CHAR> ← 일반 텍스트 <EQUATION> ← 수식 <SCRIPT>수식문법</SCRIPT> <PICTURE> ← 이미지 참조 <IMAGE BinItem="1"/> 수식 예시: "2x LEFT ( 2x+4y-1 RIGHT )" → LaTeX: "2x\left(2x+4y-1\right)" 이미지: BINDATA Base64 → 디코딩 → PNG 파일 문항 경계: CHAR 텍스트에서 "1.", "2.", "[1-2]" 패턴 탐지

Task 2 — Docker Compose 환경 구성

작업 내용

7개 서비스를 정의합니다. 설계서 v1.3의 Docker Compose 구성 + v1.4의 orchestrator 분리를 반영합니다.

핵심 포인트: orchestrator는 replicas: 1 (단일 인스턴스, 상태 정합성). worker는 replicas: 2 (병렬 처리, consumer group). backend는 API 서버. 이 3개는 같은 코드베이스에서 다른 command로 실행됩니다.
환경 변수: .env 파일에서 로드. ANTHROPIC_API_KEY, DATABASE_URL, REDIS_URL, S3_ENDPOINT, JWT_SECRET, LLM_MODE(mock/real) 등. dev 환경 기본값 사용.

완료 기준

  • docker-compose up -d → 7개 서비스 모두 healthy
  • PostgreSQL에 접속 가능 (psql 또는 pgAdmin)
  • Redis에 접속 가능 (redis-cli ping → PONG)
  • MinIO 콘솔 접속 가능 (http://localhost:9001)
  • Backend 서버 응답 (http://localhost:8000 → 아직 404여도 OK)

Task 7 — 오케스트레이터 코어

작업 내용

파이프라인의 두뇌. pipeline:results Stream을 소비하여 상태 전이를 실행하고, pipeline:tasks에 다음 작업을 발행합니다.

# 구현할 핵심 함수: # 1. handle_review_result(ref_id, stage, result) # → PASS: 다음 스테이지로 전이, 다음 에이전트 작업 발행 # → REJECT: version+1, 해당 작업팀으로 반려, reject_context 전달 # 2. dispatch_to_agent(stage, item) # → DB에서 에이전트 payload 조립 (v1.4 패치 #6 참조) # → pipeline:tasks에 발행 # 3. 3계층 상태 관리 # L1: pkey 기준, PARSING ~ L1_COMPLETED # L2-A: exam_id 기준, EXAM_COMPOSE ~ EXAM_CONFIRMED # L2-B: classroom_exam_id 기준, DEPLOY_REQUESTED ~ CLOSED # 4. PROD_REVIEW 분기 (v1.4 패치 #1) # answer_source == "ai_derived" → HUMAN_CONFIRM_ANSWER # 그 외 → DATA # 5. 예외 처리 # 타임아웃 300초, 3회 연속 반려 → HUMAN_REVIEW # EMBEDDING 실패 → 재시도 3회 → EMBEDDING_ERROR # 6. classroom_exams.status 동기화 (v1.4 패치 #4) # L2B 상태 전이마다 sync_classroom_exam_status() 호출

완료 기준

  • 더미 에이전트(dummy_agent.py)로 E2E 파이프라인 흐름 검증
  • L1: 더미 PARSING → PARSE_REVIEW(PASS) → META → ... → L1_COMPLETED
  • 반려 시나리오: PARSE_REVIEW(REJECT) → PARSING 재투입, version +1
  • pipeline_states, pipeline_history에 상태 기록 확인
  • 타임아웃 처리 동작 확인
